<h2><a href=https://www.codewars.com/kata/54ce6115975ca685dd0005d5/train/javascript target="_blank">Move-To-Front Encoding</a></h2><h3>5 kyu</h3><h2 id="motivation">Motivation</h2><p>The goal of data compression is to remove redundancy from an input. Some of you already know run-length-encoding, where elements get encoded with their number of consecutive occurences. But what if elements are not directly consecutive but just close to each other?</p><p>(This is especially important for the Burrows-Wheeler-Transformation, which creates a permutation of the input, where equal symbols are close to each other. There is also a <a href="http://www.codewars.com/kata/burrows-wheeler-transformation/" data-turbolinks="false" target="_blank">kata</a> about that from me.)</p><h2 id="move-to-front-encoding">Move To Front Encoding</h2><p>In this encoding we use a prioritized and changing alphabet for our input. As the name suggests, when encoding a symbol from our alphabet, we look up the index of that symbol in our alphabet. This is our representation for that symbol, and then change the alphabet, by moving the symbol to the front. Example with the alphabet <code>['a', 'b', 'c']</code>:</p><pre style="display: none;"><code class="language-haskell"><span class="cm-variable">encode</span> <span class="cm-string">"abc"</span> <span in the exact same direction, just that we look up the index in the alphabet, instead of the actual element.</p><h2 id="goal">Goal</h2><p>Write <code>encode</code> and <code>decode</code> for the Move-To-Front-Encoding.</p><p>Return <code>null</code> if encoding or decoding fails.<br>Also note alphabets can have repeating characters, and can be quite large.<br>The alphabet may be an Array of Numbers or a String; <code>encode</code>'s second argument will be this same type, and <code>decode</code>'s return value must be this same type.<br>Inputs must not be modified.</p>
